# Методология D.O.C.S. для Thunder Load

Единый цикл изменений, чтобы любая задача проходила одинаково: от запроса до релизной заметки.

## Расшифровка

- `D — Discover` (выявить): понять проблему, цель и пользовательскую ценность.
- `O — Organize` (организовать): зафиксировать область изменений и план.
- `C — Check` (проверить): подтвердить качество, безопасность и отсутствие регрессий.
- `S — Share` (сообщить): документировать результат для команды и пользователей.

## D — Discover

Что делаем:

- Формулируем проблему в 1-2 предложениях.
- Определяем ожидаемый пользовательский результат.
- Уточняем затронутые области: `UI`, `main`, `IPC`, `tools`, `build`, `docs`.

Артефакт этапа:

- Короткое описание задачи (issue, заметка в PR или рабочий файл).

Критерий готовности:

- Понятно, зачем изменение нужно пользователю и где именно будет правка.

## O — Organize

Что делаем:

- Разбиваем задачу на 2-5 шагов.
- Определяем риски и обратимость.
- Фиксируем, какие проверки обязательны.

Минимальный план проверок:

- Для логики: `npm run lint` и `npm test`.
- Для `templates/*.njk`: `npm run templates:build` и связанные тесты.
- Для `src/scss/*.scss`: `npm run css:build`.
- Для пользовательских изменений: обновление `whats-new.md` и `whats-new.en.md`, затем `npm run whats-new:build`.

Артефакт этапа:

- Явный список шагов и проверок перед началом правок.

Критерий готовности:

- Есть конкретный план внедрения и понятный список команд для валидации.

## C — Check

Что делаем:

- Запускаем проверки из плана.
- Делаем ручной sanity-check для ключевого пользовательского сценария.
- Проверяем diff на риски безопасности (IPC, sandbox, contextIsolation) и регрессии.

Артефакт этапа:

- Отчёт о проверках (в PR или в рабочей заметке): какие команды запускались и результат.

Критерий готовности:

- Все обязательные проверки зелёные, критичных замечаний нет.

## S — Share

Что делаем:

- Обновляем пользовательские заметки (`whats-new*.md`) при изменениях поведения приложения.
- Обновляем документацию процесса/архитектуры, если поменялись правила разработки.
- В PR кратко фиксируем: что изменилось, как проверено, какие ограничения остались.

Артефакт этапа:

- Обновлённые `whats-new*.md` (если требуется) и/или `docs/*`.

Критерий готовности:

- Любой участник команды понимает итог изменения без чтения всего diff.

## Definition of Done по D.O.C.S.

Задача считается завершённой, если:

1. Выполнены этапы `D`, `O`, `C`, `S`.
2. Обязательные команды проверки выполнены и успешны.
3. Документация и пользовательские заметки синхронизированы с фактическим поведением приложения.
