# Workflow улучшений Thunder Load

Документ описывает последовательность действий для регулярных улучшений проекта.
Цель — делать изменения предсказуемо, безопасно и проверяемо.

Базовый стандарт выполнения каждой задачи: `docs/DOCS.ru.md` (`Discover → Organize → Check → Share`).

## 1. Формулировка улучшения

- Опишите проблему/цель в 1–2 предложениях.
- Сформулируйте ожидаемый результат (что должно измениться в UX/функционале).
- Укажите область кода (UI, main‑процесс, IPC, инструменты, сборка).

## 2. Быстрая диагностика

- Проверить, где находится логика (см. `docs/WORKFLOW.ru.md`).
- Найти затронутые модули в `src/js/**` и шаблоны в `templates/**`.
- Если затрагивается UI — убедиться, что источники в `templates/` и
  потребуется `npm run templates:build`.

## 3. План изменений

- Разбить работу на 2–5 шагов.
- Отделить безопасные изменения от потенциально рискованных.
- Если затрагиваются данные пользователя — продумать миграцию/обратимость.

## 4. Реализация

- Делать правки небольшими, логическими блоками.
- Следить за безопасностью (IPC whitelist, contextIsolation, sandbox).
- Избегать CDN‑зависимостей в runtime.

## 5. Тесты и проверки

- `npm run lint` — строгий линт без предупреждений.
- `npm test` — unit‑тесты.
- При UI‑изменениях — ручная проверка ключевых сценариев.
- В тестах не допускается шумный `console.log` (используйте глобальный мок).

## 6. Документация

- Обновить `docs/WORKFLOW.ru.md`, если процесс изменился.
- Обновить корневой `whats-new.md` при изменениях пользовательских функций
  (это источник истины).

## 7. Коммиты

- Делить на смысловые коммиты (lint/рефактор отдельно от функционала).
- Сообщения коммитов: кратко, в стиле `type: summary`.

## 8. Порядок ревью

1. Автоматические проверки: `npm run lint`, `npm test`.
2. Просмотр diff: безопасность, регрессии, влияние на UX.
3. Проверка UI вручную (ключевые сценарии + доступность).
4. Проверка документации и `whatsNew.md`, если затронуты пользовательские изменения.
5. Только после этого — approve/merge.

## 9. Финальная проверка

- Пробежать глазами diff на регрессии.
- Убедиться, что `npm run lint` и `npm test` проходят.

## 10. Следующие шаги (опционально)

- Создать релизную заметку.
- Запланировать последующее улучшение.

## Список улучшений (бэклог)

1. Ввести CSP без inline‑скриптов, чтобы усилить безопасность UI.
2. Перевести динамические вставки HTML на безопасные шаблоны/санитайзер там, где есть `insertAdjacentHTML`.
3. Добавить единый лог‑хелпер для renderer (уровни, выключение в prod).
4. Улучшить UX обновлений: показать changelog из `whatsNew.md` прямо в модалке обновления.
5. Вынести список поддерживаемых платформ/хостов в конфиг, чтобы проще расширять.
6. Добавить smoke‑тест IPC: быстрые проверки ключевых каналов.
7. Авто‑проверка версии `whatsNew.md` при `npm run check`.
8. Добавить e2e‑smoke (запуск Electron + проверка главного окна).
9. Визуальная деградация при оффлайн: явный баннер “нет сети”.
10. Рефакторинг `history.js`: разбиение на под‑модули (render, pagination, filters).
11. Стабильная структура логов загрузок (schema + миграции).
12. Конфигурируемые лимиты очереди и параллельных загрузок.
13. Мини‑диагностика (CPU/RAM/диск) в “Справке”.
14. Автосборка “What’s New” из `whats-new.md` для релиза (`npm run whats-new:build`).
15. Опционально: локализация UI (RU/EN) через словари.
